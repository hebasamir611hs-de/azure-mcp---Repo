Component key
TypeMeaning
ORCHESTRATORCLAUDE.md — the router. Wears the QA Manager hat (Phase 1–2) or the SDET hat (Phase 3–3b).
SKILLA file under .claude/skills/ — a fixed, reusable procedure the orchestrator invokes.
SUB-AGENTA Claude agent (Agent tool) a skill delegates real work to — reasoning-only or coding, per its own tool grant.
MCP CALLA specific tool invocation on a registered MCP server (azure-devops, playwright, appium, figma) — the exact function name is shown on every call.

P1

Analysis & Generation

Derive full test coverage for a PBI, in chat, in the chosen analysis mode. Nothing is written to Azure in this phase.

Trigger
User submits a PBI ID or a sprint, optionally naming the mode (Normal default / Deep)
Writes to
chat only
Skills used
analyze-pbi, quick-test-cases, verify-figma-design
Sub-agents used
qa-engineer
1
orchestratorQA Manager hat
invokes (sprint mode)
mcp call — azure-devops mcp__azure-devops__get_pbis_from_sprint(iteration_path) reads every PBI in the sprint; validates readiness; auto-detects language (ar/en)
Optional entry point when given a sprint rather than a single PBI ID. A PBI is skipped only if its Description is empty — a missing Acceptance Criteria does not skip it.
2

Full Phase-1 procedure for one PBI, in the active mode. Alternative: quick-test-cases for a tight prioritized subset (happy path + top negatives + sharpest edges) — same call shape, no injection either way.

skillanalyze-pbi

The only Azure read inside analyze-pbi itself. Description-only PBIs (no AC) are a valid basis — the skill derives scope and states every assumption explicitly.
DECISION — Design Source Cascade
Fixed priority, never in parallel: a Figma link in the description/AC (step 2b) outranks a PBI image (step 2f), which outranks proceeding with no design source at all (step 2h). Exactly one of the three branches below runs.

skillanalyze-pbi

pattern matchfigma.com/(file|design)/…
No MCP call — a text scan of what step 2a already returned. Found → branch to 2c. Not found → skip to 2f.
2c
skillanalyze-pbi
invokes (Figma link found)
skillverify-figma-design
Passes the Figma URL + the PBI title/description. Waits for it to finish before deriving anything — its output feeds step 3 below, merged with whatever else the PBI gives.
2d
skillverify-figma-design
calls
mcp call — figma mcp__figma__get_figma_data(fileKey, nodeId) simplified layout/typography/color/spacing data for one node — always node-scoped, never a whole-file fetch when a node-id is available
2e
skillverify-figma-design
verifies
outputRelevance verdict (matches PBI?)
Mandatory, every time — a frame named after the PBI is not proof it is the PBI's design. Checks the node's actual content against the PBI's expected fields/copy; red flags (Lorem-ipsum body text, placeholder breadcrumbs, generically-numbered components, the same block repeated under variant labels) mean STOP and ask the human to confirm or correct the link. Live-verified on a real WOQOD file: a frame named "Contact Us" turned out to be a generic page-header template with no real form fields — this check caught it before any case was written.
2f
skillverify-figma-design
produces (on verified)
outputDesign tokens + Figma-verified UI cases
Exact copy, typography, color, and spacing per element, written as concrete assertable UI cases — feeds step 3. If the link never verifies, this step is skipped and the flow falls back to 2h's text-only path instead.
2g
skillanalyze-pbi
(no Figma link) if has_images, asks
human gate"Use attached image(s) as design?"
Metadata-driven, not automatic — covers both images pasted into Description/AC and images uploaded as attachments. No has_images → skip straight to 2h.
2h
skillanalyze-pbi
calls (only on YES)
mcp call — azure-devops mcp__azure-devops__get_story_for_analysis_with_images(story_id) downloads and returns each detected image as viewable content — vision cost incurred here, never before this explicit yes
On NO: images are neglected; the flow proceeds on text alone. If neither a Figma link nor images exist at all, analyze-pbi asks once — proceed text-only, or supply a Figma link (loops back to 2c) — before continuing.
3
skillanalyze-pbi
delegates to
sub-agentqa-engineer
Agent tool call — passes the PBI spec/AC, scope, assumptions, active mode, and whichever design source resolved above (Figma-verified UI cases, viewed images, or neither). Reasoning-only: no MCP access, no code, no file writes.
4
sub-agentqa-engineer
produces
outputTest case set (chat)
Full template-formatted cases for every in-scope category of the active mode (Normal: UI/Functional-High/Functional-Low + optional Compatibility/Auth/light Edge; Deep: all 8 categories + full 4-step edge methodology), merging in any Figma-verified UI cases from step 2f. Tags include Lifecycle (UAT/Regression) + Service + Platform + Category — never the Automation/Manual execution tag, that is a Phase-2 decision.
5
orchestratorQA Manager hat
reviews
outputTest case set (chat)
Coverage review gate against the CLAUDE.md checklist. Any missing in-scope category, vague expected result, or missing Tags value sends the set back to step 3 for another pass — not a new agent call, a resume of the same one.
6
orchestratorQA Manager hat
produces
outputQA sign-off (chat)
Categories covered, total TC count, analysis basis (AC+description / description-only, plus which design source was used — Figma-verified, images, or text-only), detected automation surface (from Platform tags), open risks/assumptions.
GATE — Sign-off Gate (human approval required)
The QA Manager's sign-off is not itself approval. The User must explicitly approve the set before Phase 2's inject-test-cases can run — no injection tool is ever called inside Phase 1.

GATE — Hat Switch
Automation begins only when the User explicitly asks (or accepts the Phase-1 lookahead offer) — never auto-triggered after injection. The orchestrator then switches from the QA Manager hat to the SDET hat.

GATE — Failures Detected (automatic, no human gate)
A failed run flows straight into Phase 3b — the one transition in the whole pipeline that needs no approval, because dedup (not a human) is what prevents duplicate bug filing downstream.
P3b

Bug Reporting on Failure

The automatic tail of any red run: triage the failures, then file or update one Bug per failure.

Trigger
run-automation completes with failures
Writes to
Azure DevOps Bug work items, .html summary report
Skills used
triage-failures, create-azure-bug, generate-summary-report
Sub-agents used
quality-control-engineer, test-failure-triage
1
orchestratorSDET hat
delegates to
sub-agentquality-control-engineer
Reads the Allure results (local filesystem — no MCP call) and extracts one structured entry per failure: test name, Test Case ID (from the traceability marker), error message, expected/actual result, screenshot path, run URL.
2
orchestratorSDET hat
invokes
skilltriage-failures
delegates to
sub-agenttest-failure-triage
Reads the Allure artifacts (results + history for retries/flakiness — no MCP call) and classifies every failed/broken test as PRODUCT_BUG, AUTOMATION_BUG, ENVIRONMENT_ERROR, or NEEDS_INVESTIGATION, each with confidence, evidence, and a suggested owner.
3
sub-agenttest-failure-triage
produces
outputBug decision (verdict + owner, per failure)
This decision informs step 5 below — it is analysis-only and never files or fixes anything itself.
4
orchestratorSDET hat
invokes (per failure)
skillcreate-azure-bug
No confirmation gate — every failure is eligible by design; dedup is the only decision. Informed by the quality-control-engineer's failure list (step 1) and the test-failure-triage decision (step 3).
4a
skillcreate-azure-bug
calls
mcp call — azure-devops mcp__azure-devops__find_existing_bug(test_case_id) dedup check — searches for an open Bug already tagged TC:<test_case_id>
4b
skillcreate-azure-bug
calls (not found)
mcp call — azure-devops mcp__azure-devops__create_bug(test_case_id, test_name, error_message, expected_result, actual_result, repro_steps, priority, screenshot_path, run_url) creates the Bug; plain-English title from actual_result; raw error under "Automation Failure Root Cause"
If found instead: mcp__azure-devops__add_bug_occurrence(bug_id, error_message, run_url) — appends and reopens from Resolved if needed. Never both, never neither.
5
mcp call resultazure-devops
produces
outputBug work item (created or updated)
Already discoverable via the query hierarchy create-bug-queries provisioned in Phase 2 — no further provisioning happens here.
6
orchestratorSDET hat
invokes (after all failures processed)
skillgenerate-summary-report
6a
skillgenerate-summary-report
calls
mcp call — azure-devops mcp__azure-devops__get_test_outcome_summary / get_test_run_outcome_summary / generate_qa_report / get_query_summary whichever supplies the test-suite outcome and/or bug/work-item payload for this run
Produces: QA summary report (.html) — pass/fail totals, bugs filed vs. updated. Read-only; never creates or edits data.
